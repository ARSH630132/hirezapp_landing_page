# GFF AI Platform: Final MVP Handoff Documentation

This document serves as the concise, practical handoff guide for deploying, running, and managing the **GFF AI Platform** (Next.js Frontend & FastAPI Backend).

---

## 1. Product Scope & Architectural Boundaries

### 1.1. Core Platform Components
The GFF AI Platform is composed of two primary layers:
1. **Next.js Frontend (App Router)**: A premium, Apple-like web interface providing client portals, administrator portals, public/educational pages, and interactive builders.
2. **FastAPI Backend (REST API)**: A lightweight, high-performance API server managing authentication (JWT), role-based access control (RBAC), multi-tenant data structures, and dashboard operations.

### 1.2. Crucial Scope Notice: GFF AI Tools
> ⚠️ **IMPORTANT DESIGN BOUNDARY**:
> All interactive **GFF AI Tools** inside the Build suite are **frontend-only, client-side mock configurations and interactive visual flows**. They are explicitly decoupled from the backend and do not call real live AI models, compile real local scripts, or connect to database persistence.
> 
> These include:
> *   **Sandbox / Agent Playgrounds** (`/build/sandbox`)
> *   **Blueprint Generator** (`/build/blueprint`)
> *   **ROI Calculator** (`/build/roi`)
> *   **Proposal Builder / Intake** (`/build/proposal`)
> *   **Agent Marketplace** (`/build/marketplace`)
> *   **AI Readiness Assessment Framework** (`/build/assessment`)
> *   **Foundry Studio** (`/build/foundry-studio`)
> *   **Interactive Assistant / Talk** (`/build/talk`)
>
> All backend API integrations are reserved strictly for operational telemetry: User Profile Management, Client Workspace Handshakes, Multi-Tenant Projects, Real-Time Invoices/Epoch Telemetry, Support Ticket Queueing, S3 Document Metadata, and Compliance Governance Controls.

## 2. Production AWS Cloud Infrastructure

To transition from local SQLite development to production, the backend is engineered for a cloud-native AWS serverless topology (FastAPI on AWS Lambda/ECS connected to DynamoDB and S3).

### 2.1. DynamoDB Data Modeling (Exactly 3 Tables)
Complex relational mappings are collapsed into a high-performance single-table design for operational workspace items (`GFF_PORTAL_ITEMS`), alongside dedicated identity tables.

| Table Name | Partition Key (PK) | Sort Key (SK) | Global Secondary Index (GSI) | Purpose |
| :--- | :--- | :--- | :--- | :--- |
| **`GFF_USERS`** | `email` (String) | *None* | *None* | Global user credentials, roles, and client ties. |
| **`GFF_CLIENTS`** | `client_id` (String) | *None* | *None* | Enterprise customer entity directory. |
| **`GFF_PORTAL_ITEMS`** | `PK` (String) | `SK` (String) | `GSI1` (`GSI1PK` & `GSI1SK`) | Multi-tenant single-table workspace store. |

#### Single-Table Entity Schema mappings (`GFF_PORTAL_ITEMS`)
1. **Projects**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `PROJ#<project_id>`
   * **`GSI1PK`**: `PROJ#<project_id>` | **`GSI1SK`**: `METADATA`
2. **AI Operations (Telemetry)**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `AIOPS#<ops_id>`
   * **`GSI1PK`**: `PROJ#<project_id>` | **`GSI1SK`**: `AIOPS#<ops_id>`
3. **Documents Metadata**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `DOC#<doc_id>`
   * **`GSI1PK`**: `PROJ#<project_id>` | **`GSI1SK`**: `DOC#<doc_id>`
4. **Invoices / Billing**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `INV#<invoice_id>`
   * **`GSI1PK`**: `BILLING_PERIOD#<year_month>` | **`GSI1SK`**: `INV#<invoice_id>`
5. **Support Tickets**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `TICKET#<ticket_id>`
   * **`GSI1PK`**: `STATUS#<status_code>` | **`GSI1SK`**: `TICKET#<ticket_id>`
6. **Governance / Controls**:
   * **`PK`**: `CLIENT#<client_id>` | **`SK`**: `GOV#<gov_id>`
   * **`GSI1PK`**: `STANDARD#<compliance_standard>` | **`GSI1SK`**: `GOV#<gov_id>`

### 2.2. Secure Object Storage (Amazon S3)
*   **Bucket Name**: `gff-portal-documents-prod`
*   **Access Paradigm**: Block All Public Access. Enforces Server-Side Encryption (SSE-KMS) and strict TLS 1.3 transport.
*   **Operation**: The frontend interacts with files using short-lived (15-minute) secure Pre-signed URLs dynamically generated by the FastAPI backend using AWS SDK (Boto3).

## 5. Execution & Build Commands Reference

### 5.1. Local Sandbox Environment Execution
Run these commands from the root of the workspace during local development:

```bash
# 1. Install Backend Dependencies (FastAPI, SQLite, SQLite-Utils, Pytest)
npm run backend:install

# 2. Setup SQLite Schema & Inject Multi-Tenant Seed Users/Items
npm run backend:seed

# 3. Launch Local Hot-Reloading FastAPI Backend (Standard Port: 8000)
npm run backend:dev

# 4. Launch Local Hot-Reloading Next.js Frontend (Standard Port: 3000)
npm run dev

# 5. Run Security, RBAC & End-to-End API Integration Smoke Tests
npm run backend:check
```

### 5.2. Production Deployment Execution
Prepare and run production workloads with these instructions:

#### Next.js Frontend Production Build
To create an optimized production bundle and boot the production web server locally or in a container:
```bash
# Build the application
npm run build

# Start the node runtime production server
npm run start
```

#### FastAPI Backend Production Launch
To boot the FastAPI server using multi-threaded Uvicorn workers in cloud hosting VMs, containers, or on-premises servers:
```bash
# Production launch command (4 workers, host bound to all network interfaces)
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```
*(For Serverless AWS Lambda deployments, configure API Gateway to route to a Lambda containing the FastAPI application wrapped with the `Mangum` ASGI adapter).*
